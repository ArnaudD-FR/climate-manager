<objective>
Add the two backend fields required by D-24 and D-25 so the frontend can bind directly without computing values from `hass.states`. (D-23: "All logic in backend".)

- **D-24:** `rooms_status[].present_person_count: int` — computed in both
  `coordinator._build_status_payload` and `websocket.ws_get_status`. Frontend
  currently intersects arrays in `room-card.ts:435` — that frontend logic is
  removed in Plan 03-12 once the backend value is in place.
- **D-25:** `get_config` response includes `climate_entities: list[str]` — the
  sorted list of all `climate.*` entity IDs in the HA entity registry. No
  frontend code currently filters `hass.states` for climate entities (the
  TRV-card section reads `roomStatus.entity_ids`), but the field is required by
  the contract so future TRV pickers and downstream UI never recompute domain
  filters in TS.

Purpose: Two backend additions in one focused plan because both touch
`websocket.py` (file ownership). Both are read-only additions to existing
responses — no new commands, no schema changes to stored data.

Output: Two new keys in WebSocket response payloads; unit-tested. </objective>
