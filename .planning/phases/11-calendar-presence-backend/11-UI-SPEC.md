## Color

All colors are HA CSS custom properties — the panel inherits the user's
active HA theme. Fixed semantic hex values are only used where HA variables
do not exist (period/presence bars per `types.ts`).

| Role            | Value / Variable                              | Usage                               |
|-----------------|-----------------------------------------------|-------------------------------------|
| Dominant (60%)  | `var(--card-background-color)`                | Card surfaces, expanded content     |
| Secondary (30%) | `var(--secondary-background-color, #f5f5f5)`  | Badges, chips background, selects   |
| Accent (10%)    | `var(--primary-color, #03a9f4)`               | Reserved elements listed below      |
| Destructive     | `var(--error-color, #f44336)`                 | chip-remove hover only              |

Accent (`var(--primary-color)`) is reserved for:

1. The active tab-btn underline in the week-switcher (even/odd toggle)
2. The `select:focus` border (2px on keyboard focus)
3. The `reset-btn` border and text color
4. `force-present` mode badge border and text

New calendar mode badge: uses the same secondary/muted style as the
`scheduled` badge — `background: var(--secondary-background-color)`,
`color: var(--secondary-text-color)`. No new accent usage.

New calendar period color: add `"calendar": "#5C6BC0"` (indigo-400) to
`PRESENCE_COLORS` in `types.ts`. This is a fixed semantic hex (same
convention as existing presence/period colors). Also add
`"calendar": "C"` to `PERIOD_LABELS` and `"calendar": "Calendar"` to
`PERIOD_DISPLAY_NAMES`.

## Component Inventory

All new UI elements are built from existing Lit component patterns. No new
third-party components.

### New: Calendar Config Block (person-card.ts)

Rendered inside the person card's expanded state when
`currentMode === "calendar"`. Positioned ABOVE the room associations section
(per D-14).

Structure (top to bottom):

1. Section label: "Calendar source" — uses `.section-label` style
2. Calendar entity picker: native `<select class="mode-select">` populated
   from `Object.keys(hass.states).filter(id => id.startsWith("calendar."))`
   — friendly name shown as option text, entity_id as value. Placeholder
   option: "— Select a calendar —" with value `""`.
3. Event means selector: native `<select class="mode-select">` with two
   options:
   - "Absent during events" (value `"absent"`, default selected)
   - "Present during events" (value `"present"`)
   Section label above: "Event means"
4. Pre-heat lead time input: native `<input type="number">` with
   `min="0"` `max="480"` `step="5"`. Label above: "Start heating X min
   before return" (section label style). Suffix span with text "min" placed
   inline after the input. Matches the existing `<input type="number">`
   pattern established in the HA 2026.x memory constraint.

### New: Calendar mode badge (person-card.ts)

Add branch `case "calendar": return { cls: "calendar", text: "Calendar" }`
in `_getBadgeInfo()`. CSS class `.mode-badge.calendar` uses same styling
as `.mode-badge.scheduled` (muted secondary colors).

### New: Calendar option in mode select (person-card.ts)

Add `<option value="calendar">Calendar</option>` to the presence mode
`<select>`.

### New: Calendar period state in time-bar / period editor

When `state === "calendar"` is selected for a period in Scheduled mode:
- Period block renders with the new `"#5C6BC0"` color from `PRESENCE_COLORS`.
- Single-char label `"C"` from `PERIOD_LABELS`.
- Full label `"Calendar"` from `PERIOD_DISPLAY_NAMES`.
- Inline calendar config block (entity picker + event_means select, same
  as the top-level calendar config block but scoped to the period) appears
  below the period row inside the period editor.

### Layout Reorder (person-card.ts) — D-14

New card-content section order when expanded:

1. "Presence mode" section label + mode select (unchanged)
2. Hint paragraph / stuck-mode warning (unchanged)
3. **Calendar config block** (visible only when `mode === "calendar"`)
4. **Schedule section** — "Presence schedule" label + schedule-type select
   + time-bar (visible only when `mode === "scheduled"`)
5. "Room associations" section label + chips + search-picker (moved below)

## Interaction Contract

### Auto-save on change (D-09, phase 9 pattern)

All new controls save immediately on `change` event — no Save button:

- Calendar entity `<select>` → `@change` → `setPersonConfig({ calendar_config: { entity_id, event_means } })`
- Event means `<select>` → `@change` → `setPersonConfig({ calendar_config: { entity_id, event_means } })`
- Lead time `<input>` → `@change` (on blur) → `setPersonConfig({ preheat_lead_minutes: value })`

### Calendar entity picker population

Populated at render time from `hass.states`. Filter:
`id.startsWith("calendar.")`. Show `hass.states[id].attributes.friendly_name
?? id` as display text. If no `calendar.*` entities exist, show a single
disabled option: "No calendar entities found in Home Assistant."

### Conditional rendering

- Calendar config block: visible only when `this.config?.mode === "calendar"`.
- Period inline calendar config: visible only when `period.state === "calendar"`.
- Lead time input: visible only within the calendar config block (same
  visibility as the block).

### Input validation — lead time

- Type: number, min 0, max 480.
- On invalid input (non-integer, out of range): do not call save; leave the
  field as-is. Browser native constraint validation is sufficient (no custom
  error UI needed for this phase).

### No new loading states

Calendar entity list is synchronously derived from `hass.states` — no async
fetch, no loading spinner needed. HA pushes `hass.states` updates to the
panel; the select repopulates on next render.
