<objective>
Implement the `ClimateManagerCoordinator` control loop and wire it into the integration lifecycle. The coordinator polls every minute, derives the active period from the current wall-clock time via `dt_util.now()`, resolves the desired temperature per global mode (Off / Time program / Time program & presences) using the Plan 01 schedule engine, and pushes to TRVs only on change — with a manual-override hold and an immediate startup push for restart recovery.

Purpose: This is the integration's spine. It turns the pure decision logic from Plan 01 into actual TRV commands at the right moment, DST-safe by construction (re-derives from `dt_util.now()` every tick — no pre-scheduled boundaries), and cleans up its listener on unload (Pitfall 1).

Output: New `coordinator.py`; `__init__.py` extended (dataclass fields + setup/unload wiring); `test_coordinator.py` integration tests; `test_init.py` extended for the unload-cancels-scheduler case.
</objective>

| Boundary | Description |
|----------|-------------|
| HA event loop → coordinator | async_track_time_interval fires async_evaluate every minute with a UTC datetime; coordinator must not trust that argument as local time. |
| stored config → coordinator | runtime_config (global_mode, period_temperatures, programs, persons) read each tick. Originates from Phase 3 panel (validated by validate_7day_coverage at save time). |
| HA state machine → coordinator | hass.states.get(entity_id).attributes["temperature"] read for manual-override detection — value may be None or non-float. |
| coordinator → climate.* services | Outbound: set_hvac_mode + set_temperature via set_trv_temperature. Trusted internal call; availability-guarded in trv.py. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
