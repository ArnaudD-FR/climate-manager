<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HA Store → in-memory config dict | Disk-resident JSON (potentially edited by user or malformed by failed writes) crosses into the integration's runtime config. Sparse-merge over DEFAULT_CONFIG is the only sanitiser. |
| In-memory config dict → HA Store (async_save) | Code anywhere in the integration may pass a config dict to async_save. validate_zone_assignment is the last guard before disk. |
| const.py module import → all callers | DEFAULT_CONFIG is a process-wide singleton; mutation at import time would propagate everywhere. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | HA Store JSON on disk (storage.py async_load) | accept | HA Store file is inside `.storage/` which is owned by the HA process user; any attacker with write access can already compromise the entire HA install. Out of scope for Phase 4. |
| T-04-02 | Information Disclosure | Zone names in default_zone_name and zones[*].name | accept | Zone names are user-provided display labels with no secret content. No PII risk per project scope (household config only). |
| T-04-03 | Tampering | Mutation of DEFAULT_CONFIG at import time (Pitfall 1 in RESEARCH.md) | mitigate | const.py exposes `zones: {}` as a literal — never populated. `async_load` calls `copy.deepcopy(DEFAULT_CONFIG)` (existing line 53/59). Test `test_load_fresh_install_returns_copy_not_same_object` (existing) covers this. No new code mutates DEFAULT_CONFIG. |
| T-04-04 | Tampering | Save of malformed zone assignment (room references non-existent zone, or two rooms share a zone_id) | mitigate | `validate_zone_assignment(config)` invoked from `async_save` before `_store.async_save` raises `ValueError` with explicit message. Test `test_save_rejects_unknown_zone_id` and `test_validate_zone_assignment_duplicate_zone_id_raises` enforce this. |
| T-04-05 | Repudiation | Silent data loss when loading v1.0 stored data without zones | mitigate | Sparse-merge over DEFAULT_CONFIG already preserves all v1.0 keys (existing test_load_sparse_stored_data_merges_over_defaults). New test `test_load_v10_data_without_zones_gets_defaults` proves zones/default_zone_name fall back to defaults without overwriting stored values. STORAGE_VERSION stays at 2 (D-04) so HA Store never invokes migration callbacks. |
| T-04-06 | Denial of Service | Validation helper hangs or recurses on malicious config | accept | validate_zone_assignment is O(n) in number of rooms with bounded set comparison. No recursion. Number of rooms is bounded by HA area registry (realistic max ~100). No DoS vector. |
| T-04-07 | Elevation of Privilege | A zone_id collision with a future sentinel value | mitigate | D-07 mandates UUIDs (Phase 5 will call `uuid.uuid4()`). Phase 4 documents the convention via `import uuid` in storage.py. validate_zone_assignment treats zone_id as opaque string — no sentinel comparison. |
| T-04-SC | Tampering | npm/pip installs in this plan | accept | No new packages installed in Phase 4 (research §"No New Packages"). uuid and copy are stdlib. Package Legitimacy Audit not applicable. |
</threat_model>
