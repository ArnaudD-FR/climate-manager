<domain>
## Phase Boundary

Extend the storage schema to support heating zones. Add `zones: {}` and `default_zone_name` to `DEFAULT_CONFIG`. Add optional `zone_id` to room entries. No coordinator logic, no WebSocket commands, no frontend changes — storage layer only.

</domain>

<decisions>
## Implementation Decisions

### Default Zone Identity
- **D-01:** The Default Zone has NO storage entry. `zone_id` absent or `null` on a room means "belongs to the Default Zone". There is no sentinel ID, no `is_default` flag, no entry in `zones: {}`.
- **D-02:** The Default Zone's mode and time program ARE the existing `global_mode` and `global_time_program` fields. No new fields for zone mode/schedule. The UI simply presents these under the Default Zone tab.
- **D-03:** The Default Zone's name is stored as a new top-level field `"default_zone_name"` in DEFAULT_CONFIG (e.g., `"default_zone_name": "Home"`). Absent in v1.0 data → sparse-merge gives the default name automatically.

### Schema Versioning
- **D-04:** STORAGE_VERSION is NOT bumped. All schema changes are purely additive (`zones: {}` and `default_zone_name` added to DEFAULT_CONFIG). The existing sparse-merge loading in `async_load` handles new top-level keys transparently — v1.0 stored data loads cleanly, missing keys fall back to DEFAULT_CONFIG defaults. No HA Store migration risk.

### Room zone_id Storage Convention
- **D-05:** Zone membership is stored as `zone_id` on the room entry (foreign key on room, not `room_ids` on zone). Consistent with research finding — zone deletion only needs to scan rooms and clear their `zone_id`.
- **D-06:** When a room is in the Default Zone, `zone_id` is NOT stored (sparse model — D-11 preserved). A room entry only includes `zone_id` if the room belongs to a custom zone.
- **D-07:** Custom zone IDs are UUIDs, generated by the backend at zone creation time. Stable, collision-free, stored as strings.

### Folded Todos
- **"Define multiple heating zones with independent modes"** (pending todo, planted 2026-05-26) — This todo seeded the entire v1.1 milestone. Phase 4 delivers the storage foundation for it; Phases 5 and 6 deliver the evaluation engine and UI.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ClimateManagerStore.async_load()` (`storage.py:38`) — sparse-merge over `DEFAULT_CONFIG`. Adding `"zones": {}` and `"default_zone_name": "Home"` to DEFAULT_CONFIG is sufficient for v1.0 compatibility; no changes to the merge loop needed.
- `copy.deepcopy(DEFAULT_CONFIG)` pattern — used for fresh installs and as the merge base. New keys are included automatically once added to DEFAULT_CONFIG.

### Established Patterns
- **Sparse storage (D-11):** Only non-default values are written. Room entry only appears if it has custom config. `zone_id` on a room follows this: only present if room is in a custom zone.
- **Manual migration in async_load:** v1.0 had person mode rename migrations written inline in `async_load`. If any zone migration logic is needed in future (none for Phase 4), this is where it goes.
- **Dict keying:** Rooms keyed by HA area_id (string). Persons keyed by `person.*` entity_id. Custom zones keyed by UUID string.

### Integration Points
- `DEFAULT_CONFIG` in `const.py` is the single source of truth for schema shape and defaults. Phase 4 only touches this file and adds TypeScript type stubs (if the team wants early type safety in `types.ts`).
- `STORAGE_VERSION` in `const.py` stays at `2` — no change to `storage.py` constructor call.

</code_context>

<specifics>
## Specific Ideas

- Default zone name field key: `"default_zone_name"`, default value: `"Home"` (or a localized equivalent — TBD at implementation time).
- Zone object shape in `zones: {}`:
  ```python
  "zones": {
      "<uuid>": {
          "name": "<string>",
          "mode": "<global_mode_value>",       # same enum as global_mode
          "time_program": { <daily_program> }   # same structure as global_time_program
      }
  }
  ```
- Room entry extension (only when in a custom zone):
  ```python
  "<area_id>": {
      "zone_id": "<uuid>",          # only present if room is in a custom zone
      "room_mode": "global"|"custom"|"frost_protection",
      "time_program": { ... }        # only if room_mode == "custom"
  }
  ```
