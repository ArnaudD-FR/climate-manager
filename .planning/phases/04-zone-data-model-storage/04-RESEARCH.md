## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** The Default Zone has NO storage entry. `zone_id` absent or `null` on
  a room means "belongs to the Default Zone". There is no sentinel ID, no
  `is_default` flag, no entry in `zones: {}`.
- **D-02:** The Default Zone's mode and time program ARE the existing
  `global_mode` and `global_time_program` fields. No new fields for zone
  mode/schedule.
- **D-03:** The Default Zone's name is stored as a new top-level field
  `"default_zone_name"` in DEFAULT_CONFIG (e.g., `"default_zone_name": "Home"`).
  Absent in v1.0 data → sparse-merge gives the default name automatically.
- **D-04:** STORAGE_VERSION is NOT bumped. All schema changes are purely
  additive (`zones: {}` and `default_zone_name` added to DEFAULT_CONFIG). The
  existing sparse-merge loading in `async_load` handles new top-level keys
  transparently. No HA Store migration risk.
- **D-05:** Zone membership is stored as `zone_id` on the room entry (foreign
  key on room, not `room_ids` on zone).
- **D-06:** When a room is in the Default Zone, `zone_id` is NOT stored (sparse
  model). A room entry only includes `zone_id` if the room belongs to a custom
  zone.
- **D-07:** Custom zone IDs are UUIDs, generated by the backend at zone creation
  time. Stable, collision-free, stored as strings.

## Summary

Phase 4 is a narrow, low-risk schema extension. The integration already has a
stable sparse-merge Store pattern (`storage.py::ClimateManagerStore.async_load`)
where adding new keys to `DEFAULT_CONFIG` automatically makes them available to
existing installs without any migration code.

The two additions to `DEFAULT_CONFIG` in `const.py` are: (1)
`"default_zone_name": "Home"` — a top-level scalar picked up transparently by
existing v1.0 data; and (2) `"zones": {}` — a top-level dict following the same
sparse-storage convention as `"rooms"` and `"persons"`. The zone object shape
mirrors the existing global fields: `name` (string), `mode` (same enum as
`global_mode`), `time_program` (same structure as `global_time_program`).

ZONE-04 (single-zone enforcement) is the only piece of new logic: a validation
helper that rejects a save if a `zone_id` value appears on more than one room
entry, or if a room carries a `zone_id` that references a non-existent zone. All
nine existing `test_storage.py` tests pass against the current codebase; new
tests for the zone schema and ZONE-04 guard will follow the identical `hass`
fixture pattern.

**Primary recommendation:** Add two keys to `DEFAULT_CONFIG`, define the zone
object shape in `const.py` documentation, add `zone_id` to the room schema
comment block, add a `validate_zone_assignment` helper to `storage.py`, and
extend `types.ts` with `ZoneConfig` and `RoomConfig.zone_id`. No other files
change in Phase 4.

### Pattern 1: Additive DEFAULT_CONFIG Extension

**What:** Add new top-level keys to `DEFAULT_CONFIG` in `const.py`. The existing
sparse-merge loop in `async_load` handles them automatically — no changes to
`storage.py`'s merge logic.

**When to use:** Whenever a new top-level schema key is introduced that has a
sensible default (empty dict, empty string, scalar). All existing stored data
automatically receives the default on next load.

**Example (from existing codebase — adapted for zones):**

```python
# Source: custom_components/climate_manager/const.py (existing pattern)
DEFAULT_CONFIG: dict = {
    "version": STORAGE_VERSION,
    "global_mode": DEFAULT_GLOBAL_MODE,
    "period_temperatures": dict(DEFAULT_PERIOD_TEMPERATURES),
    "global_time_program": copy.deepcopy(_DEFAULT_DAILY_PROGRAM),
    "rooms": {},
    "persons": {},
    # Phase 4 additions:
    "default_zone_name": "Home",   # D-03: Default Zone display name
    "zones": {},                   # ZONE-01: custom zones dict, keyed by UUID string
}
```

### Pattern 2: Sparse Room Entry Extension

**What:** `zone_id` is only written to a room entry when the room belongs to a
custom zone. When `zone_id` is absent from a room entry, the runtime interprets
the room as belonging to the Default Zone. Consistent with D-11 (sparse storage)
and D-06.

**When to use:** Any optional per-room field whose absence has a well-defined
default meaning.

**Example:**

```python
# Room in a custom zone (zone_id present):
"rooms": {
    "living_room": {
        "zone_id": "a1b2c3d4-...",   # UUID string — D-07
        "room_mode": "global",
    }
}

# Room in Default Zone (zone_id absent — D-06):
"rooms": {
    "bedroom": {
        "room_mode": "custom",
        "time_program": { ... }
    }
}
```

### Pattern 3: ZONE-04 Validation Helper

**What:** A pure-function helper that inspects the in-memory config dict before
it is persisted, ensuring no `zone_id` value appears on more than one room and
that every `zone_id` references an existing entry in `zones`.

**When to use:** Called immediately before
`await self._store.async_save(config)`.

**Example:**

```python
# Source: new helper in storage.py
def validate_zone_assignment(config: dict) -> None:
    """Raise ValueError if zone assignment invariants are violated.

    ZONE-04: a room belongs to at most one zone (guaranteed by data structure —
    area_id keys are unique — but zone_id values must each appear only once
    across all room entries if that constraint is desired).
    Also validates that every zone_id references an existing zone.
    """
    zones = config.get("zones", {})
    seen_zone_ids: set[str] = set()
    for area_id, room_cfg in config.get("rooms", {}).items():
        zone_id = room_cfg.get("zone_id")
        if zone_id is None:
            continue
        if zone_id not in zones:
            raise ValueError(
                f"Room '{area_id}' references unknown zone_id '{zone_id}'"
            )
        # Note: the structural guarantee of dict keying means each area_id is
        # unique, so one room cannot appear twice. ZONE-04 is structurally
        # satisfied. The validation below is a belt-and-suspenders check for
        # any caller that builds zone_id lists differently in Phase 5.
        if zone_id in seen_zone_ids:
            raise ValueError(
                f"zone_id '{zone_id}' assigned to multiple rooms — ZONE-04 violated"
            )
        seen_zone_ids.add(zone_id)
```

**Implementation note:** ZONE-04 says "a room can belong to at most one zone".
Because rooms are stored as a dict keyed by `area_id`, each room physically
appears exactly once in storage — structural uniqueness is already enforced. The
validation helper's real job is to ensure every `zone_id` reference points to a
real zone (referential integrity), and optionally to catch logic bugs where a
caller mistakenly assigns the same zone to overlapping room sets from a list
perspective.

### Anti-Patterns to Avoid

- **Storing Default Zone as a sentinel entry:** D-01 explicitly prohibits a
  `"default"` key in `zones: {}`. Doing so would break v1.0 compatibility
  (sparse-merge would overwrite `{}` with a populated dict on a fresh v1.0 load
  if the sentinel weren't in stored data).
- **Bumping STORAGE_VERSION:** D-04 prohibits bumping to 3. HA Store treats a
  version mismatch as a signal to run a migration callback; if no callback is
  registered, HA may discard stored data. Additive-only changes do not need a
  bump.
- **Writing `zone_id: null` explicitly:** D-06 prohibits storing `null` for
  Default Zone members. The sparse model requires omission, not null-writing, to
  remain consistent with how `rooms: {}` works for all other default values.
- **Caching zone names on room entries:** Like area.name and person
  friendly_name, zone display names must NOT be duplicated onto room entries.
  Read from `zones[zone_id].name` at display time (prevents stale-name bugs when
  zones are renamed in Phase 5).

## Common Pitfalls

### Pitfall 1: Mutating DEFAULT_CONFIG at Import Time

**What goes wrong:** `DEFAULT_CONFIG["zones"]["some-uuid"] = {...}` in any
module that imports from `const.py` corrupts the global default, causing all
subsequent `copy.deepcopy(DEFAULT_CONFIG)` calls on fresh installs to return
polluted data.

**Why it happens:** `DEFAULT_CONFIG` is a module-level dict. Python module
import caches it as a singleton. Any mutation affects all future deep copies.

**How to avoid:** Never mutate `DEFAULT_CONFIG` directly. Always work on the
result returned by `async_load()` (which is already a deep copy).

**Warning signs:** Test `test_load_fresh_install_returns_copy_not_same_object`
fails; fresh-install tests return unexpected keys in `zones`.

### Pitfall 2: Sparse-Merge Overwrites Nested Dicts Wholesale

**What goes wrong:** The merge loop in `async_load` uses
`result[key].update(value)` for nested dicts. For `"zones"`, this means stored
zones are merged key-by-key into the default `{}` — correct. But if a future
caller saves only a partial zones dict (e.g., after deleting a zone), a naive
merge could resurrect deleted zones from DEFAULT_CONFIG if
`DEFAULT_CONFIG["zones"]` were non-empty.

**Why it happens:** `dict.update()` adds keys, never removes them. The sparse
model relies on `DEFAULT_CONFIG["zones"]` being `{}` so that stored data is the
authoritative source.

**How to avoid:** `DEFAULT_CONFIG["zones"]` MUST remain `{}`. Never populate it
with example zones. The stored file is the authoritative zones list;
DEFAULT_CONFIG provides only the empty-dict fallback.

**Warning signs:** A deleted zone reappears after restart.

### Pitfall 3: Zone Object Shape Mismatch with Phase 5

**What goes wrong:** Phase 5 writes zone objects to storage using a shape the
coordinator does not expect (e.g., `time_program` stored as a list instead of a
dict keyed by day).

**Why it happens:** Zone `time_program` reuses the same structure as
`global_time_program` and room `time_program`, but this is only documented in
CONTEXT.md — not enforced anywhere in Phase 4 code.

**How to avoid:** Document the canonical zone object shape in a comment block in
`const.py` immediately next to the `DEFAULT_CONFIG` definition (same pattern as
the existing rooms sub-schema comment at line 107). Phase 5 implementors will
see it before writing any CRUD code.

**Warning signs:** Coordinator crashes with `KeyError` when iterating zone time
programs in Phase 5.

### Pitfall 4: TypeScript RoomConfig.zone_id Not Optional

**What goes wrong:** `types.ts` adds `zone_id: string` (required) to
`RoomConfig`, causing TypeScript compile errors in Phase 6 for every existing
room entry that has no zone_id.

**Why it happens:** Sparse storage means most rooms will not have `zone_id` in
the JSON. The TypeScript type must match.

**How to avoid:** Declare `zone_id?: string` (optional, not
`zone_id: string | null`) — consistent with the sparse model. The frontend
interprets absent `zone_id` as Default Zone membership.
