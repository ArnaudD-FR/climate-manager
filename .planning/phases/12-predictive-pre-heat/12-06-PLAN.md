<objective>
Move the pre-heat enable/disable flag from per-room
(`RoomConfig.preheat_enabled`) to per-zone scope, on the Python backend. Closes
GAP-01 from 12-HUMAN-UAT.md: during UAT (recorded in 12-05-SUMMARY.md) the user
confirmed all rooms in a zone should share a single pre-heat on/off setting
(PREHEAT-01). `preheat_max_lead_minutes` STAYS per-room (PREHEAT-01 — per the
gap scope in the planning context). The pre-heating status line and suppression
warning (PREHEAT-04) are unaffected in meaning — only the gate source changes.

Purpose: The toggle granularity shipped wrong (12-04). This plan corrects the
backend contract — the coordinator read source, a new zone WS command, and a
storage migration — so the frontend plan (12-07) can wire the zone toggle and
remove the room toggle against a stable contract.

Output: Zone-scoped `preheat_enabled` read by the coordinator, a new
`set_zone_preheat` WS command, a one-time storage migration, updated tests.

## Default Zone decision (DOCUMENTED)

The Default Zone has NO entry in `config.zones` — it is synthesized from
`global_mode` / `global_time_program` (see coordinator `_resolve_zone_config`),
and its display name lives in the top-level `default_zone_name` key.
`ws_rename_zone` already routes the `"default"` sentinel to that top-level key.

**Decision: Option A.** Add a top-level `default_zone_preheat_enabled?: boolean`
key to the config, mirroring `default_zone_name`. Rooms with no `zone_id` read
`config.get("default_zone_preheat_enabled", False)`.

Rationale — this is the only option consistent with the existing
Default-Zone-as-top-level-key pattern:
- Option C (sentinel UUID "default" inside `config.zones`) would make the
  Default Zone appear in zone enumeration and break `_resolve_zone_config` and
  `validate_zone_assignment` invariants — explicitly avoided by T-05-01.
- Option B (Default Zone always pre-heat-disabled) would deny pre-heat to the
  most common room placement (rooms with no custom zone), defeating PREHEAT-01.
</objective>
