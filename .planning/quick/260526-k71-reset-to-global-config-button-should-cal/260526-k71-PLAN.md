<objective>
Move the "Reset to global configuration" schedule-copy logic from the frontend into a new backend WebSocket command, then simplify the frontend handler to a dumb send-and-reload pattern.

Purpose: Schedule manipulation belongs on the backend, where runtime_config is the source of truth. Today's frontend reads `global_time_program` from the cached `panelConfig`, deep-copies it in JS, and sends it back via `set_room_config` — this is a multi-step round-trip that can drift from backend state and duplicates logic that already exists for global resets. The new pattern (single WS command that mutates server-side, frontend just refreshes) matches the existing `reset_time_program` / `reset_period_temperatures` handlers.

Output:
- New WS command `climate_manager/reset_room_to_global_program` (payload `{room_id}`) that deep-copies the current `global_time_program` into `rooms[room_id].time_program` and sets `rooms[room_id].room_mode = "custom"`.
- New `WsClient.resetRoomToGlobalProgram(roomId)` wrapper.
- `_onResetToGlobal` reduced to: send WS command → `panel.reloadConfig()` → toast.
- Pytest coverage and rebuilt `panel.js` bundle.
</objective>
